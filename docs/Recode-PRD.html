Recode

RE the run.

Recode is a lightweight, standalone, cross-platform coding-agent runtime with dual surfaces: a human-first TUI and an automation-friendly CLI for OpenClaw. Its core value is not chat polish. It is visible, policy-driven execution built for re-plan, re-try, re-route, re-sume, and re-code with explicit retry, timeout, approval, and audit boundaries.

2. Background & Objectives

Build a coding runtime that feels sharp in the terminal, stays safe under enterprise expectations, ships as Linux and Windows binaries, and exposes a stable CLI contract so OpenClaw can call it directly during real coding workflows.

2.1 Philosophy: The Meaning of “RE”

RE is the philosophy of disciplined return.

Good systems do not pretend they are always right on the first pass. They observe, adapt, recover, and continue. Recode is built on that belief.

  • Re-plan when reality invalidates the original path
  • Re-try when failure is temporary but the objective remains valid
  • Re-route when the workflow must take a different course
  • Re-sume when interruption should not mean loss of progress
  • Re-code when the implementation must evolve under evidence

Recode is not named for repetition. It is named for resilience, revision, and controlled recovery. The runtime exists to make iteration explicit, governable, and trustworthy.

2.2 Problem Statement

Existing coding-agent tools are often strong at conversation but weak at execution control. In company environments, missing controls around retry, timeout, approvals, scheduling, and structured automation make them awkward or unsafe to use at scale.

  • Retry behavior is opaque or inflexible
  • Timeouts are too coarse or not operator-visible
  • Automation interfaces are unstable or absent
  • Cross-platform support breaks on shell and PTY assumptions
  • Security posture is weaker than what practical teams need
  • Dynamic replanning and workflow steering are not first-class

7. Workflow Orchestration Requirements

Recode must provide dynamic workflow control comparable in class to Claude Code's flexible coding flow, but with more explicit operator visibility and policy control.

  • Dynamic replanning: revise execution path when tool output, tests, or new context changes the situation.
  • Conditional branching: next steps may depend on outputs, validation results, approval outcomes, or policy checks.
  • Operator intervention: users can pause, redirect, skip, retry, or force a branch without corrupting session state.
  • Approval-aware flow: risky actions create explicit approval nodes rather than ad-hoc prompts.
  • Checkpoint and resume: workflow state resumes from the last stable checkpoint after interruption or failure.
  • Skill-driven orchestration: skills define workflow templates, step types, and policy defaults without arbitrary runtime mutation.
  • Scheduled execution: workflows can launch immediately, later, on intervals, or through cron expressions.
  • Structured outcome routing: success, failure, timeout, refusal, or cancellation each map to explicit next-state behavior.

9. Non-Functional Requirements

9.1 Performance & Reliability

  • Initial structured response for routine CLI state queries should return within 2 seconds on a normal local environment.
  • Scheduled jobs and session state must survive process restarts through persistent local state.
  • Crash recovery must preserve enough state to resume or inspect the last stable checkpoint.

9.2 Platform Support

  • Ship release binaries for Linux and Windows.
  • No dependency on tmux.
  • No dependency on bash or POSIX-only semantics.
  • Process, PTY, shell, and cancellation behavior must be abstracted.
  • Windows behavior must be first-class, not “best effort later.”

9.3 Security & Data Handling

Security target: comparable class to OpenCode and Claude Code.

  • Default-safe local execution posture
  • Explicit approval gates for risky or destructive actions
  • Tool allowlists and constrained invocation policy
  • Clear separation between trusted instructions and external content
  • Auditable logs for tool use, approvals, retries, scheduled runs, and failures
  • No hidden autonomous background behavior outside declared workflow

10. User Flow & Edge Cases

10.1 Core User Flow

  1. User or OpenClaw starts a task or scheduled run.
  2. Runtime creates or resumes a session and loads policy + skill context.
  3. Workflow plans initial subgoals and selects the first executable step.
  4. Step executes through LLM/tool/approval path with timeout and retry policy.
  5. Runtime evaluates result, then continues, replans, blocks for approval, or fails visibly.
  6. User or OpenClaw inspects state, steers if needed, and resumes until completion or explicit stop.

10.2 Key Edge Cases

ScenarioExpected Handling
LLM timeoutMark attempt timed out, apply retry policy, log cause, and expose state to TUI/CLI.
Tool exits non-zeroRecord failure class, allow retry/replan/manual steer based on policy.
Approval deniedTransition to blocked or refused state, never silently continue.
Empty stateTUI/CLI must show explicit no sessions / no jobs / no history states, not blank screens.
Invalid inputReturn structured validation errors with clear field-level reason where applicable.
Process restart during scheduled waitRestore scheduled jobs from persistent state on next startup.
User cancel / back outMove workflow to canceled or paused state without corrupting session history.

11. Architecture Direction

Recommended

  • Language: Rust
  • TUI: Ratatui
  • Async runtime: Tokio
  • Execution model: Session actor + Task graph + Step runner + Attempt policy
  • Workflow control: dynamic branching, replanning, approval nodes, checkpoint/resume
  • LLM integration: provider abstraction with OpenAI-compatible endpoint support and Codex API support
  • Skill model: Declarative-first + limited hooks
  • Scheduler: built-in cron and delayed-run subsystem over the shared runtime core
  • Surfaces: TUI frontend + CLI frontend over shared core
